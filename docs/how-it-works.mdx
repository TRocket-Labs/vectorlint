VectorLint reviews target files against your content rules and reports specific, evidence-backed findings.

If you haven't run your first review yet, start with the [Quickstart](/quickstart). This page explains what happens when you run that command.

## The review pipeline

Every time you run `vectorlint doc.md`, VectorLint follows the same high-level path.

### 1. Resolve targets and rules

VectorLint reads your project configuration to determine which rule packs apply to each target file. It loads rule files, applies `VECTORLINT.md` as global review context when present, and builds source-backed rules.

If no `.vectorlint.ini` exists, VectorLint detects `VECTORLINT.md` and creates a synthetic style-guide rule automatically. This is the zero-config path.

### 2. Choose a review approach

VectorLint uses `auto` by default and selects an approach based on the content being reviewed. You can override the default with `--model-call` when a particular document needs a different approach.

See [Model calls](/model-calls) for selection guidance and CLI examples.

### 3. Review the content

VectorLint reviews the target against each applicable rule. For long documents, it divides the content into manageable sections and combines the results.

### 4. Validate and score findings

Before reporting a finding, VectorLint confirms that its quoted evidence appears in the target content. It removes duplicate findings and calculates a score from the verified violations.

Findings without locatable evidence are omitted from the reported violations.

### 5. Present the results

VectorLint presents the verified findings using the requested output format:

- `line` for human-readable terminal output
- `json` for VectorLint's native machine-readable output
- `vale-json` for Vale-compatible integrations
- `rdjson` for reviewdog-compatible integrations
